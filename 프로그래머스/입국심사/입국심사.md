해당 문제의 경우 파라매트릭 서치에 해당되는 문제로서 난이도가 있어 보였으나, 오랜 시간을

들여서 풀어보고 싶어서 도전해보았습니다.

먼저, 사람의 수 n, 각각의 심사관들이 입국심사를 할때 소요되는 시간이 들어있는 배열 times

두개의 파라미터만 주어지고, 이를 통해서 모든 사람이 최소한의 시간으로 입국심사를 마치는

시간을 answer 변수에 삽입하여 반환하는 것이 목적이었습니다.

먼저, left, middle, right를 교재의 힌트를 참고하여 long 타입으로 선언하였고,

이를 계산하는 식을 코드로 작성하였습니다.

먼저 left는 1로,

right의 경우, 시간의 최대값을 삽입하여야 합니다. 하지만 이때 정확한 최대값을 구하기 위해서는

주어지는 파라미터를 이용해야 했으며, 단순하게 사람의 수 \* 배열 중 최대값 으로 지정하기로

생각하고 먼저 배열을 오름차순으로 정렬하였고, times[times.length - 1]이 해당 배열의

최대값이며, 여기에 사람의 수 n을 곱한, times[times.length - 1] \* n 을 right의 값으로

지정하였습니다.

이를 통해 (left + right) / 2 로 middle 값을 삽입하였고,

n, times, middle을 compare 함수의 파라미터로 지정하였습니다.

또한 compare 함수에서는 64비트 count 변수를 선언하여,

times.length 동안 middle 즉, 입국심사에 소모되는 시간 / 최소 심사관의 심사시간으로

나누게 되면 해당 값은 시간 내에 몇 명을 심사할 수 있는지를 도출하게 되고,

해당 값을 count에 더해가면서 count가 n, 즉 주어진 사람의 수의 이상이라면,

compare 함수는 종료되고 count 값을 반환하도록 작성하였습니다.

위와 같이 작성하고 코드 실행을 해보았으나 예상한 값과 아예 다르게 나왔고 전체 코드의

작동 방식을 다시 검토해보니 아예 compare 함수가 반환해야 하는 값이 count가 아닐 것이라는 생각이

들게 되었습니다. 애초에 제가 원하는 반환값은 주어진 사람수를 최소한으로 심사할 수 있는 시간

이지만, compare 에서 사람 수를 반환하고 있기에 이것이 잘못되었다고 생각했고, compare 에서

반환하는 값이 양수, 음수인 즉 1과 -1로만 반환하여야 하고, 이를 Solution 에서 검사하는

구조가 맞을 것이라고 생각하였습니다.

위의 구조로 작성한 코드는

반환값이 1이라면, 해당 값이 정답인지를 middle의 값을 1 줄여서 다시 compare 함수를

호출해보고 해당 결과가 -1인지, 1인지를 검사하게 됩니다.

이때 -1이라면, 현재의 middle 값이 정답이라는 말이며, 바로 반복문을 중지시켜서, middle의

값 즉, 주어진 인원을 심사하는데 걸리는 최소시간을 반환하게 됩니다.

하지만 1이라면 아직 시간이 원하는 정답이 아니라는 말이기에, middle의 값을 1 줄이게 됩니다.

또한 반환값이 -1이라면, 시간이 부족하다는 의미기에, middle의 값을 1 증가시켜서

반복문을 다시 수행하게 됩니다.

위처럼 작성한 코드를 실행해보니, 테스트는 성공이었습니다.

하지만 해당 방식이 시간초과를 유발한다는 것을 알게 되었습니다.

이를 수정하기 위해서 middle의 값을 조절하는 것이 아니라 탐색의 범위인

left, right 를 직접 조절하는 방식을 사용하면 시간 초과를 방지할 수 있다는 사실을 알게

되었습니다. 이는 최소 시간을 구하는 과정에서, 전체 탐색 범위를 줄여서 탐색 시간을 줄이고

반환값이 -1이라면 시간이 부족하다는 것이기에 left의 값을 1 증가시키고

반환값이 1이라면 시간이 남는다는 것이기에 right를 1 감소하여 탐색 범위를 줄입니다.

해당 과정이 반복되면서 탐색 범위가 정답에 가까워지고, 만약 middle의 값이 현재

정답이라면, 조건문을 통해서 1이 줄어들고 다시 검사하여 -1이 반환되면, 범위를 1 증가시키고

left가 right보다 커지게 되어 while문이 종료되게 되고, 이때 최소시간을 반환하게 됩니다.

위 구조로 코드를 작성하여 코드를 실행해보니 정상적으로 실행은 되나,

정확도가 50퍼센트를 조금 넘는 결과를 확인할 수 있었고 이는 times[times.length-1]를 계산하는

과정에서 오버플로우가 발생하여 오차가 발생하는 문제임을 깨닫게 되었습니다.

이를 해결하기 위해서 수식의 앞에 (long)을 붙여서 크기를 늘려주고 코드를 제출하니

정상적으로 작동하는 모습을 확인할 수 있었습니다.
